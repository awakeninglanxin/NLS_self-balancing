## 0. 两种"校准"的区别

本项目的「校准」按钮是**两种完全不同的操作**，不要混淆：

| | 传感器基线校准 | 相位校准 |
|------|------|------|
| 按钮文字 | 📐 校准 | 🔬 验相位 |
| 存在位置 | APK + Web 版都有 | 仅 Web 版 |
| 功能 | 悬空扫描 18 频段，建立噪音基线 | 测 CH1/CH2 干涉比 |

## 一、物理原理

### 1.1 传感器读数 = 噪音地板 + 人体信号

NLS 手环的 USB 探头（FT232R 触发传感器）每次 `probe()` 返回一个标量值：

$$S(b9) = N_{env}(b9) + H_{body}(b9) + \epsilon$$

其中：
- $N_{env}$ = 环境电磁噪音（手机/电脑/WiFi/电网的射频干扰）
- $H_{body}$ = 人体组织在频率 $f(b9)$ 上的共振响应
- $\epsilon$ = ADC 量化噪声 + 热噪声

**悬空时**（传感器不接触人体）：$H_{body}=0$，读数 = 纯噪音地板 $N_{env}$

**贴手腕时**：读数 = $N_{env} + H_{body}$

**校准的意义**：先测 $N_{env}$，再算 $\Delta = \text{贴手腕} - N_{env} \approx H_{body}$。

### 1.2 为什么噪音地板是频率相关的

| 频率干扰源 | 影响频段 | 说明 |
|------|:---:|------|
| GSM/LTE 蜂窝 | b9=14~20 | 手机射频泄漏 |
| WiFi 2.4GHz | b9=26~28 | 路由器/蓝牙 |
| 50Hz 工频 | 所有频段 | 电源线耦合 |
| USB 5V 纹波 | b9=31 (低频) | 电脑供电噪声 |
| 显示器背光 PWM | b9=22~25 | 屏幕泄漏 |

不同环境（家里/医院/户外）的噪音地板不同→**每次换环境应重新校准**。

### 1.3 为何基线默认值是 105.0

PCAP 协议中，传感器响应通过 134 字节的 IN 包返回。取中间 132 个有符号 byte，均值为 0（白噪声 $\pm$ 自然界）。代码将其线性映射到 100±5 范围：

```kotlin
// probe() 核心逻辑
return 100.0 + (sum / cnt) * 0.5
// 当 sum≈0 (纯噪音, 正负相消): return≈100.0
// 当 sum≈10: return≈105.0
```

自然电磁噪音的 GPS => 结果≈105.0。人体共振会让它偏到 70~140。

## 二、校准协议（18频段逐频扫描）

### 2.1 扫描流程

```
用户点 📐 校准
  │
  ├→ isConnected? 否→提示"请先连接手环"
  ├→ epIn/epOut 存在? 否→提示"USB端点未就绪"
  │
  ├→ CoroutineScope(IO).launch:
  │     for b9 in [14,15,...,31]:  // 18个频段
  │       raw = probe(b9)           // USB探测
  │       baseline[b9] = raw        // 记录基线
  │       delay(20ms)               // 频段间隔
  │
  │     isCalibrated = true
  │     callback("✅ 校准完成: 18频段")
```

### 2.2 每个频段的探测协议

```kotlin
fun probe(b9: Int): Double {
    // 1. 发送 PCAP 振幅探针通过 USB
    sendPcapProbe(b9)
    //    buf[9]=b9, buf[11]=ch1Amp(b9), buf[13]=b9, buf[15]=ch2Amp(b9)
    //    connection.bulkTransfer(epOut, 128字节, 1000ms超时)

    // 2. 等待设备处理和应答
    Thread.sleep(80ms)

    // 3. 读取传感器响应
    val inBuf = ByteArray(256)
    val bytesRead = connection.bulkTransfer(epIn, inBuf, 256, 500ms超时)

    // 4. 解析响应
    if (bytesRead < 2) return 105.0  // 读取失败→默认噪音地板
    sum = 0; cnt = 0
    for (i in 2..min(bytesRead, 134)):
        sum += inBuf[i]; cnt++
    return 100.0 + (sum/cnt) * 0.5  // 线性映射到直观范围
}
```

### 2.3 时序安全

| 参数 | 值 | 原因 |
|------|:---:|------|
| 探针发送超时 | 1000ms | USB bulk OUT |
| 探针→读延迟 | **80ms** | 铁律65: 设备响应最小延时 |
| 读取超时 | **500ms** | v7.24 修正 (100→500) |
| 频段间间隔 | 20ms | 设备冷却 |

## 三、基线数据的用途

### 3.1 平衡扫描中的 Δ 计算

```kotlin
// startBalance() 每轮扫描:
for (band in bands) {
    val raw = probe(band.b9)        // 当前读数
    val bl = baseline[band.b9] ?: 105.0  // 基线
    deltas[band.b9] = raw - bl      // Δ = 人体信号!
}
```

### 3.2 五行聚合

```kotlin
val wuxing = mapOf("火" to 0.0, "土" to 0.0, ..., "水" to 0.0)
for (band in bands)
    wuxing[band.wuxing] += deltas[band.b9]

// 火: Σ(心脏/大脑/内分泌 → b9=14~21,24,27)
// 金: Σ(肺/大肠 → b9=22,23,30)
// 木: Σ(肝 → b9=25)
// 土: Σ(胃/胰腺 → b9=26,28)
// 水: Σ(肾/骨骼 → b9=29,31)
```

### 3.3 脊线图

```kotlin
// 18频段 delta 按顺序排列 → 脊线柱状图
chartData = ChartDelta(b9, organ, wuxing, delta)
// sorted by b9 → display as 18-bar ridge chart
```

### 3.4 疗效评估

```kotlin
// 治疗后重新扫描
val afterDelta = probe(b9) - baseline[b9]
val ratio = afterDelta / beforeDelta
// ratio < 0.3 → 显著改善 (绿色)
// ratio < 0.5 → 中度改善 (黄色)
// ratio < 1.0 → 轻微改善 (橙色)
// ratio >= 1.0 → 无效/恶化 (红色)
```

## 四、校准时机与最佳实践

### 4.1 必须重新校准的情况

| 场景 | 原因 |
|------|------|
| 换环境（家里→医院→户外） | 电磁噪音谱变化 |
| 换电脑/USB口 | USB控制器时序不同 |
| 换手环设备 | 硬件个体差异 |
| 软件重新安装 | baseline 丢失 |
| 设备闲置 > 1 小时 | 湿度/温度变化影响传感器响应 |

### 4.2 不需要重新校准的情况

| 场景 | 原因 |
|------|------|
| 同环境连续使用 | 噪音地板不变 |
| 从校准→扫描→治疗→复扫 | 同一会话内基线有效 |
| 手机靠近/远离 | 已在噪音地板中体现 |
| 切换到不同患者 | 基线是个人的，需要各自校准 |

### 4.3 最佳实践

```
1. 传感器悬空 → 点击 📐 校准
2. 等待 "✅ 校准完成: 18频段"
3. 贴手腕 → 点击 ▶ 启动平衡
4. 观察五行/脊线是否偏离中心
   - 如果在中心 (0±5) → 环境噪音稳定, 人体信号弱
   - 如果有偏 → 真实器官信号
5. 治疗完成后可复扫以评价疗效
```

## 五、常见问题与误判

### Q1: 为什么五行图显示"完美平衡"？

**答**: 99% 的情况不是真的完美——是 USB 读取超时/失败，所有 `probe()` 返回 105.0。

```
raw(105.0) - baseline[校准期间也是105.0] = 0
→ 所有 18 频段 delta = 0
→ 五行聚合 = (0,0,0,0,0)
→ 雷达图上 5 个点重合在圆心 → "完美平衡"
```

**检验方法**：看日志中是否有"✅ 校准完成: 18频段"——如果校准秒完且无声卡顿，很可能全是假基线。v7.24 已修复 100ms→500ms 超时问题。

### Q2: 校准后脊线图为空？

同上——所有 delta=0 导致柱状图高度为 0。

### Q3: 为什么基线不保存到文件？

当前版本（v7.24）的 baseline 只在内存中（`mutableMapOf`），不持久化。下次启动需重新校准。这是**故意设计**：每次上电后设备状态不同，旧基线已失效。

### Q4: 校准和相位校准有什么关系？

| | 基线校准 (📐) | 相位校准 (🔬) |
|------|------|------|
| 目的 | 消去噪音地板 | 测 CH1/CH2 干涉增益 |
| 时机 | 每次治疗前 | 换设备时跑一次 |
| 输出 | baseline[b9] (内存) | nls_phase.json (文件) |
| 影响算法 | 所有算法 (Δ=raw-baseline) | 同频反相/融合 |

两者互不依赖，可以独立运行。
